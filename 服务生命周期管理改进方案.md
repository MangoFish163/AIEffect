## 2. 现有问题识别

### 2.1 严重问题 (必须解决)

| 问题           | 风险等级 | 影响               | 发生场景       |
| -------------- | -------- | ------------------ | -------------- |
| **僵尸进程**   | 🔴 极高  | 端口占用、内存泄露 | 主进程异常退出 |
| **强制终止**   | 🔴 极高  | 数据丢失、DB损坏   | 每次关闭       |
| **无健康检查** | 🔴 高    | 服务崩溃无感知     | 运行时         |
| **依赖不可靠** | 🟡 中    | 服务启动顺序错乱   | 启动时         |
| **无运行模式区分** | 🔴 高 | 开发/生产环境行为不一致 | 部署时    |
| **日志分散**   | 🟡 中    | 问题排查困难       | 调试时         |
| **数据库并发风险** | 🔴 高 | 数据损坏、死锁    | 多服务访问时   |

### 2.2 具体问题分析

#### 问题 1: 僵尸进程风险

```powershell
# 当前代码
$logServiceProcess = Start-Process -FilePath "powershell" ...
```

**风险**:

- PowerShell 启动嵌套 PowerShell 再启动 Python
- 如果顶层 PowerShell 被杀死，中间层可能残留
- Python 进程成为孤儿进程，继续持有端口

**后果**:

- 端口 8501/8502/8505 被占用
- 下次启动失败
- 需要手动 `taskkill /F /IM python.exe`

#### 问题 2: 强制终止导致资源泄露

```powershell
# 当前代码
Stop-Process -Id $apiGatewayProcess.Id -Force
```

**风险**:

- 不发送 SIGTERM，直接强制终止
- SQLite 连接未关闭 → 数据库损坏风险
- 临时文件未清理
- WebSocket 连接未正常关闭

#### 问题 3: 无健康检查与自动恢复

**风险**:

- 服务崩溃后用户无感知
- 需要手动重启整个应用
- 没有服务状态监控

#### 问题 4: 依赖管理不可靠

```powershell
# 当前使用固定等待
Start-Sleep -Seconds 2
```

**风险**:

- 2 秒可能不够（系统负载高时）
- 2 秒可能浪费（系统空闲时）
- 没有验证依赖服务是否真正就绪

#### 问题 5: 无运行模式区分 (严重遗漏)

**当前问题**:

- 开发模式使用 `npm run dev` 启动前端，生产模式使用打包后的静态文件
- 两种模式下的进程管理方式完全不同
- 没有统一的环境配置管理

**后果**:

- 开发环境正常但生产环境异常
- 前端资源路径不一致
- 调试信息泄露到生产环境

#### 问题 6: 日志分散无聚合

**当前问题**:

- 各服务日志独立输出到不同位置
- 没有统一的日志级别控制
- 跨服务问题难以关联分析

**后果**:

- 问题排查需要查看多个日志文件
- 无法快速定位问题根源
- 日志文件占用磁盘空间无限制

#### 问题 7: 数据库并发访问风险

**当前问题**:

- Log Service 和 API Gateway 都访问 SQLite 数据库
- 没有连接池管理
- 长时间运行可能出现锁竞争

**后果**:

- 数据库死锁导致服务无响应
- 数据写入冲突
- 数据库文件损坏

## 3. 改进目标

### 3.1 核心目标

1. **零僵尸进程**: 主进程退出时，所有子服务必须完全终止
2. **优雅关闭**: 服务有 10 秒时间清理资源
3. **自动恢复**: 服务崩溃后 5 秒内自动重启
4. **依赖可靠**: 按正确顺序启动，验证就绪状态
5. **运行模式明确**: 区分开发/生产模式，行为一致可预测
6. **日志统一聚合**: 集中收集、查询和分析
7. **数据库安全访问**: 避免并发冲突和数据损坏

### 3.2 非功能性目标

| 指标         | 目标值  | 当前值      |
| ------------ | ------- | ----------- |
| 服务启动时间 | < 10 秒 | ~6 秒       |
| 优雅关闭超时 | 10 秒   | 0 秒 (强制) |
| 故障检测时间 | < 15 秒 | 无          |
| 自动重启时间 | < 5 秒  | 无          |
| 僵尸进程概率 | 0%      | > 30%       |
| 日志检索时间 | < 3 秒  | 无          |
| 数据库并发冲突 | 0 次   | 偶发        |

#### 4.2.4 关闭流程设计

```
Rust ServiceManager
       │
       ▼
┌──────────────┐
│ 1. 发送关闭信号 │ ──────► 设置全局 shutdown_flag
│   (广播)      │
└──────────────┘
       │
       ▼
┌──────────────┐
│ 2. 反向关闭服务 │ ──────► 按启动相反顺序关闭
│   (依赖倒序)   │         WebSocket → API Gateway → Log Service
└──────────────┘
       │
       ▼
┌──────────────┐     成功      ┌──────────────┐
│ 3. 优雅关闭   │ ───────────► │ 4. 清理完成   │
│   (单个服务)  │              │              │
└──────────────┘              └──────────────┘
       │
       │ 超时/失败
       ▼
┌──────────────┐
│ 5. 强制终止   │ ──────► kill -9 (SIGKILL)
│              │
└──────────────┘
```

**详细关闭步骤**:

1. **HTTP 通知** (首选): `POST http://localhost:{port}/shutdown`
   - 服务收到后自行清理并退出
   - 超时: 5 秒

2. **信号发送** (备选): `SIGTERM` (Unix) / `Ctrl+C` (Windows)
   - 触发 Python 信号处理器
   - 超时: 3 秒

3. **强制终止** (最后手段): `kill -9` / `TerminateProcess`
   - 不保证资源清理
   - 立即执行

#### 4.3.2 拓扑排序算法

```rust
async fn resolve_dependencies(&self) -> Result<Vec<String>, Error> {
    // Kahn 算法实现拓扑排序
    let mut in_degree: HashMap<String, usize> = HashMap::new();
    let mut graph: HashMap<String, Vec<String>> = HashMap::new();

    // 构建图
    for service in &self.services {
        in_degree.entry(service.name.clone()).or_insert(0);
        for dep in &service.depends_on {
            graph.entry(dep.clone()).or_default().push(service.name.clone());
            *in_degree.entry(service.name.clone()).or_insert(0) += 1;
        }
    }

    // Kahn 算法
    let mut queue: VecDeque<String> = in_degree.iter()
        .filter(|(_, &deg)| deg == 0)
        .map(|(name, _)| name.clone())
        .collect();

    let mut result = Vec::new();
    while let Some(name) = queue.pop_front() {
        result.push(name.clone());
        if let Some(dependents) = graph.get(&name) {
            for dep in dependents {
                let deg = in_degree.get_mut(dep).unwrap();
                *deg -= 1;
                if *deg == 0 {
                    queue.push_back(dep.clone());
                }
            }
        }
    }

    Ok(result)
}
```

#### 4.5.2 日志收集方式

**方式 1: HTTP 上报 (推荐)**

```python
# Python 服务中的日志处理器
import logging
import requests

class AggregatedLogHandler(logging.Handler):
    def __init__(self, aggregator_url: str, service_name: str):
        super().__init__()
        self.aggregator_url = aggregator_url
        self.service_name = service_name
        
    def emit(self, record):
        log_entry = {
            "timestamp": datetime.utcnow().isoformat(),
            "service": self.service_name,
            "level": record.levelname,
            "message": self.format(record),
            "source": f"{record.filename}:{record.lineno}",
        }
        try:
            requests.post(
                f"{self.aggregator_url}/api/logs/ingest",
                json=log_entry,
                timeout=1
            )
        except:
            pass  # 日志上报失败不应影响主服务
```

```
           │  (SQLite)   │

```

**风险**:
- SQLite 在并发写入时性能下降严重
- WAL 模式在高并发下仍可能出现锁超时
- 数据库文件损坏风险

#### 4.6.2 解决方案: 数据库访问代理

```
┌──────────────┐      ┌──────────────┐
│ Log Service  │      │ API Gateway  │
│              │      │              │
│ HTTP Client  │      │ HTTP Client  │
└──────┬───────┘      └──────┬───────┘
       │                     │
       └──────────┬──────────┘
                  ▼
       ┌─────────────────────┐
       │  Database Proxy     │
       │  (Log Service 内部) │
       │                     │
       │  ┌───────────────┐  │
       │  │ Connection    │  │
       │  │ Pool (单连接) │  │
       │  └───────┬───────┘  │
       └──────────┼──────────┘
                  ▼
           ┌─────────────┐
           │ aieffect.db │
           └─────────────┘
```

**实现方式**:

Log Service 作为唯一的数据库写入者，其他服务通过 HTTP API 访问数据：

```python
# API Gateway 中不再直接访问数据库
# 改为调用 Log Service API

async def query_logs_via_service(filters: dict) -> List[LogEntry]:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "http://localhost:8505/api/logs/query",
            json=filters
        )
        return response.json()
```

## 5. Windows 平台特殊处理

### 5.1 Windows 平台风险评估

| 风险项 | 严重性 | 说明 | 当前方案缺陷 |
|--------|--------|------|--------------|
| **信号处理机制差异** | 🔴 极高 | Windows 不完全支持 POSIX 信号 | 原方案依赖 SIGTERM |
| **进程组管理** | 🔴 高 | Windows 进程组行为与 Unix 不同 | 原方案使用 Unix 进程组概念 |
| **控制台信号** | 🔴 高 | `GenerateConsoleCtrlEvent` 限制多 | Ctrl+C 信号传递不可靠 |
| **作业对象 (Job Object)** | 🟡 中 | 需要显式配置 JOB_OBJECT 才能级联终止 | 原方案未考虑 |
| **路径处理** | 🟡 中 | Windows 路径长度限制、空格问题 | 可能导致启动失败 |
| **权限问题** | 🟡 中 | 某些端口/操作需要管理员权限 | 未明确处理 |
| **守护进程概念** | 🟡 中 | Windows 无守护进程概念 | 服务后台化方式不同 |

### 5.2 Windows 平台详细风险分析

#### 风险 1: 信号处理机制差异

**问题描述**:
- Windows 不支持 POSIX `SIGTERM`、`SIGKILL` 等信号
- Python 的 `signal` 模块在 Windows 上功能受限
- `Ctrl+C` (`SIGINT`) 是唯一可靠支持的信号

**原方案问题**:
```rust
// 原方案假设可以使用 SIGTERM
process.kill_with(Signal::SIGTERM);
```

**在 Windows 上此代码会失败或行为异常**。

#### 风险 2: 进程组级联终止失效

**问题描述**:
- Unix: 杀死进程组会终止组内所有进程
- Windows: 默认情况下子进程不会随父进程终止

**原方案问题**:
```powershell
# PowerShell 启动的 Python 进程成为独立进程
Start-Process python -ArgumentList "service.py"
# 杀死 PowerShell 不会终止 Python
```

**后果**: 僵尸进程产生，端口被占用。

#### 风险 3: 控制台信号传递限制

**问题描述**:
- `GenerateConsoleCtrlEvent` 只能发送到同一控制台的进程组
- 需要创建新的控制台窗口才能发送信号
- GUI 应用（Tauri）默认没有控制台

**原方案问题**:
```rust
// 在 GUI 应用中此操作会失败
ctrl_c_event(pid);
```

### 5.3 Windows 平台解决方案

#### 5.3.1 三级关闭策略 (Windows 优化版)

```rust
#[cfg(target_os = "windows")]
mod windows_shutdown {
    use windows_sys::Win32::System::Threading::{OpenProcess, TerminateProcess};
    use windows_sys::Win32::Foundation::{CloseHandle, HANDLE};
    
    /// Windows 平台关闭策略
    pub async fn shutdown_process(pid: u32, timeout_ms: u32) -> Result<(), ShutdownError> {
        // 第 1 级: HTTP 优雅关闭 (最可靠)
        if let Ok(_) = http_shutdown(pid, timeout_ms / 2).await {
            return Ok(());
        }
        
        // 第 2 级: 发送 Ctrl+C (需要附加控制台)
        if let Ok(_) = send_ctrl_c(pid, timeout_ms / 4).await {
            return Ok(());
        }
        
        // 第 3 级: 强制终止
        force_terminate(pid)
    }
    
    fn force_terminate(pid: u32) -> Result<(), ShutdownError> {
        unsafe {
            let handle: HANDLE = OpenProcess(0x0001, 0, pid); // PROCESS_TERMINATE
            if handle.is_null() {
                return Err(ShutdownError::ProcessNotFound);
            }
            
            let result = TerminateProcess(handle, 1);
            CloseHandle(handle);
            
            if result != 0 {
                Ok(())
            } else {
                Err(ShutdownError::TerminateFailed)
            }
        }
    }
}
```

#### 5.3.2 作业对象 (Job Object) 防止僵尸进程

```rust
#[cfg(target_os = "windows")]
mod job_object {
    use windows_sys::Win32::System::JobObjects::*;
    use windows_sys::Win32::Foundation::HANDLE;
    
    /// 创建作业对象，确保子进程随父进程终止
    pub struct JobObject {
        handle: HANDLE,
    }
    
    impl JobObject {
        pub fn new() -> Result<Self, std::io::Error> {
            unsafe {
                let handle = CreateJobObjectW(std::ptr::null_mut(), std::ptr::null());
                if handle.is_null() {
                    return Err(std::io::Error::last_os_error());
                }
                
                // 配置作业对象：当最后一个句柄关闭时终止所有进程
                let mut info: JOBOBJECT_EXTENDED_LIMIT_INFORMATION = std::mem::zeroed();
                info.BasicLimitInformation.LimitFlags = JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE;
                
                let result = SetInformationJobObject(
                    handle,
                    JobObjectExtendedLimitInformation,
                    &info as *const _ as *const _,
                    std::mem::size_of::<JOBOBJECT_EXTENDED_LIMIT_INFORMATION>() as u32,
                );
                
                if result == 0 {
                    return Err(std::io::Error::last_os_error());
                }
                
                Ok(JobObject { handle })
            }
        }
        
        pub fn assign_process(&self, process_handle: HANDLE) -> Result<(), std::io::Error> {
            unsafe {
                let result = AssignProcessToJobObject(self.handle, process_handle);
                if result == 0 {
                    return Err(std::io::Error::last_os_error());
                }
                Ok(())
            }
        }
    }
    
    impl Drop for JobObject {
        fn drop(&mut self) {
            unsafe {
                CloseHandle(self.handle);
            }
        }
    }
}
```

#### 5.3.3 Windows 服务启动封装

```rust
#[cfg(target_os = "windows")]
pub struct WindowsProcessManager {
    job: JobObject,
    processes: HashMap<String, Child>,
}

#[cfg(target_os = "windows")]
impl WindowsProcessManager {
    pub fn new() -> Result<Self, std::io::Error> {
        Ok(WindowsProcessManager {
            job: JobObject::new()?,
            processes: HashMap::new(),
        })
    }
    
    pub async fn spawn_service(&mut self, config: &ServiceConfig) -> Result<u32, ProcessError> {
        use std::os::windows::process::CommandExt;
        
        // Windows 特定：使用 CREATE_NEW_PROCESS_GROUP 标志
        const CREATE_NEW_PROCESS_GROUP: u32 = 0x00000200;
        
        let mut cmd = Command::new(&config.command);
        cmd.args(&config.args)
            .current_dir(&config.working_dir)
            .envs(&config.env_vars)
            .creation_flags(CREATE_NEW_PROCESS_GROUP)
            .stdout(Stdio::piped())
            .stderr(Stdio::piped());
        
        let mut child = cmd.spawn()?;
        
        // 将进程加入作业对象
        if let Some(raw_handle) = child.raw_handle() {
            self.job.assign_process(raw_handle as _)?;
        }
        
        let pid = child.id();
        self.processes.insert(config.name.clone(), child);
        
        Ok(pid)
    }
}
```

```rust
pub fn sanitize_windows_path(path: &str) -> String {
    // 处理 Windows 长路径问题 (MAX_PATH = 260)
    let path = if path.len() > 240 && !path.starts_with("\\\\?\\") {
        format!("\\\\?\\{}", path)
    } else {
        path.to_string()
    };
    
    // 处理空格路径：使用引号包裹

    } else {
        path
    }
}
```

   - 开发环境可创建 `.development` 标记文件

## 11. 决策记录

### 11.1 方案选择理由

**为什么选择 Rust 而不是其他方案?**

1. **进程管理能力**: Rust 提供精细的进程控制，包括信号处理、进程组管理
2. **跨平台支持**: 一套代码支持 Windows、macOS、Linux
3. **Tauri 集成**: 与 Tauri 框架无缝集成，共享 IPC 机制
4. **性能**: 相比 Node.js/Electron，资源占用更低
5. **类型安全**: 编译时捕获错误，减少运行时问题

**为什么选择 HTTP /shutdown 而不是信号?**

1. **Windows 兼容性**: Windows 信号支持有限
2. **可靠性**: HTTP 请求可确认服务收到关闭指令
3. **灵活性**: 服务可以自定义关闭逻辑
4. **可扩展性**: 易于添加关闭原因、超时等参数

**为什么选择 Log Service 作为数据库唯一写入者?**

1. **避免并发冲突**: 消除 SQLite 并发写入问题
2. **简化架构**: 单一数据源，责任清晰
3. **可维护性**: 数据库逻辑集中在一处
4. **性能**: HTTP 本地调用延迟可接受 (< 1ms)
